## Création d'images : la méthode naïve

<Note type="warning" title="Trois cas d'usage seulement">
  - Pour un usage pédagogique.
  - Lorsque l'on n'est pas en mesure de configurer un environnement autrement qu'en mode graphique (Eclipse, par exemple).
  - Pour déboguer une construction automatisée (`docker build`).
</Note>

### Utilisation de docker commit

On modifie un conteneur, puis on fige sa couche en lecture/écriture dans une nouvelle image :

```console
jmc@laptop ~ $ docker run -it -h conteneur --name conteneur_avant_commit ubuntu /bin/bash
root@conteneur:/# echo "hello world !" > /nouveau_fichier_dans_le_systeme_de_fichier_racine.txt
root@conteneur:/# exit
```

```console
jmc@laptop ~ $ docker images
REPOSITORY  TAG     IMAGE ID      CREATED      SIZE
ubuntu      latest  7b9b13f7b9c0  2 weeks ago  118 MB
jmc@laptop ~ $ docker commit conteneur_avant_commit image_apres_commit_du_conteneur
sha256:ca0028fa88252c281db33655b17517f3484b82ae42057e26193a6b91045bf2b8
jmc@laptop ~ $ docker images
REPOSITORY                        TAG     IMAGE ID      CREATED         SIZE
image_apres_commit_du_conteneur   latest  ca0028fa8825  16 minutes ago  118 MB
ubuntu                            latest  7b9b13f7b9c0  2 weeks ago     118 MB
```

La nouvelle image contient bien la modification :

```console
jmc@laptop ~ $ docker run --rm image_apres_commit_du_conteneur cat /nouveau_fichier_dans_le_systeme_de_fichier_racine.txt
hello world !
```

<Note type="warning" title="Problème de résolution DNS dans certains sous-réseaux">
  Il se peut que votre réseau vous oblige à passer par un DNS précis, qui ne correspond pas aux adresses présentes dans le `/etc/resolv.conf` de l'hôte (surcharge effectuée par NetworkManager ou systemd-resolved). Dans ce cas, la résolution de noms ne se fera pas dans le conteneur (et vous ne pourrez pas installer de paquets), car la surcharge n'affecte pas le conteneur, qui s'en tient au fichier `/etc/resolv.conf`. Il vous faut soit changer le contenu de `/etc/resolv.conf`, soit passer l'argument `--dns IP_DNS` au lancement du conteneur. Pour obtenir `IP_DNS`, vous pouvez utiliser `nmcli device show | grep DNS` ou `resolvectl status`.
</Note>

## Création d'images : le Dockerfile

### La méthode efficace

- Un **Dockerfile** est un document texte contenant toutes les commandes que vous exécuteriez normalement manuellement sur un OS (apt/yum, configuration de logiciels, création d'utilisateurs, variables d'environnement…).
- Avec la commande `docker build`, Docker crée des images **automatiquement** en lisant les instructions du Dockerfile.
- À **chaque** instruction du Dockerfile, `docker build` génère une image — et une couche si nécessaire (car potentiellement vide). On parle d'image **intermédiaire**, sauf pour la dernière instruction, qui correspond à l'image finale (*top level image*).
- Les images intermédiaires sont générées par un mécanisme de **commit** des conteneurs dans lesquels les instructions ont été exécutées.
- On peut créer plusieurs images avec un seul Dockerfile (mais on ne devrait pas).

<Note type="info" title="Évolution · le builder (BuildKit)">
  - **Avant.** `docker build` utilisait le constructeur historique : une image (et un conteneur committé) par instruction, étapes strictement séquentielles — c'est ce que montrent les sorties de terminal de ce cours.
  - **Le problème.** Pas de parallélisme entre instructions indépendantes, un cache grossier, et des secrets (clés, jetons) qui finissaient gravés dans les couches s'ils transitaient par un `ARG`.
  - **Aujourd'hui.** Depuis Docker 23.0 (2023), **BuildKit** est le constructeur par défaut (`docker buildx`) : graphe d'étapes parallélisé, cache fin (`RUN --mount=type=cache`), secrets éphémères (`--mount=type=secret`), build multi-architecture. Le modèle couches + cache reste conceptuellement identique, mais les images intermédiaires n'apparaissent plus dans `docker images -a`.
  - **Pourquoi c'est mieux.** Des builds plus rapides et reproductibles, des secrets non exposés, une meilleure granularité de cache. Pour reproduire les sorties historiques de ce cours : `DOCKER_BUILDKIT=0 docker build .` (constructeur en voie de suppression).
</Note>

### La notion de contexte

- Il faut **dédier un répertoire** à chaque création d'image par Dockerfile. En effet, `docker build` fonctionne avec une notion de **contexte** qui s'applique récursivement à partir du chemin du répertoire contenant le Dockerfile.
- À l'instar de git, vous pouvez utiliser un fichier `.dockerignore` : `docker build` ignorera tous les fichiers et répertoires qui y sont indiqués (ils ne feront pas partie du contexte).
- Les commentaires dans un Dockerfile sont préfixés par `#`.

<Note type="danger" title="Ne jamais créer un Dockerfile à la racine du système">
  Le contexte envoyé au démon Docker concernerait l'intégralité du système de fichiers !
</Note>

### Les instructions

Il est possible d'utiliser des **variables d'environnement** (comme `HOME`, `PATH`…) interprétables par certaines instructions, marquées ✦ ci-dessous :

| Instruction | Rôle |
|---|---|
| `FROM` | image de base |
| `RUN` | exécuter une commande (nouvelle couche) |
| `CMD` | commande par défaut du conteneur |
| `LABEL` ✦ | métadonnées (`MAINTAINER` est déprécié au profit de `LABEL`) |
| `EXPOSE` ✦ | documenter un port d'écoute |
| `ENV` ✦ | définir une variable d'environnement |
| `ADD` ✦ / `COPY` ✦ | copier des fichiers du contexte dans l'image |
| `ENTRYPOINT` | exécutable du conteneur |
| `VOLUME` ✦ | déclarer un point de montage |
| `USER` ✦ | utilisateur des instructions suivantes |
| `WORKDIR` ✦ | répertoire de travail des instructions suivantes |
| `ARG` | variable de construction |
| `ONBUILD` ✦ | instruction différée aux images enfantes |
| `STOPSIGNAL` ✦ | signal d'arrêt |
| `HEALTHCHECK` | sonde de santé |
| `SHELL` | shell des instructions `RUN` |

### Exemple : le Dockerfile d'un serveur ssh

`/home/jmc/sshd/Dockerfile` :

```dockerfile
# Image de départ
FROM debian:testing-slim

# Variables d'environnement
ENV VERSION=1 \
    DEBIAN_FRONTEND=noninteractive

# Métadonnées de l'image
LABEL maintainer="Jean-Mathieu Chantrein" \
    maintainer_email="jean-mathieu.chantrein@univ-angers.fr" \
    version="${VERSION}"

# Création d'un utilisateur lambda
RUN useradd lambda --create-home --shell /bin/bash && \
    echo "lambda:password" | chpasswd

# Installation des paquets, suppression du cache apt
RUN apt update && \
    apt install -yq openssh-server && \
    rm -rf /var/lib/apt/lists/* && \
    mkdir /var/run/sshd

# setuid sur le démon ssh
RUN chmod u+s /usr/sbin/sshd

# root => lambda
USER lambda

# Commande par défaut au lancement du conteneur
CMD ["/usr/sbin/sshd", "-D"]
```

<Note title="ENV DEBIAN_FRONTEND : à ne pas reproduire">
  Déclarer `DEBIAN_FRONTEND=noninteractive` en variable d'environnement **persistante** (instruction `ENV`) est une mauvaise pratique — elle restera active dans tous les conteneurs et toutes les images enfantes ([ticket GitHub](https://github.com/moby/moby/issues/4032)). Préférez la portée d'une seule commande : `RUN DEBIAN_FRONTEND=noninteractive apt install …`. Le [TP](../tp-docker-wordpress/) y revient.
</Note>

### Encore quelques informations

- Il est important de taguer/versionner ses images de production.
- Les couches les plus volumineuses devraient être les plus basses (utilisation efficace du mécanisme de cache).
- Les instructions susceptibles de changer régulièrement devraient se situer dans les couches **hautes**.
- Attention aux directives `update` et `install` lors de l'utilisation du cache : un `RUN apt update` servi par le cache peut dater.
- Tout changement de contexte (modification d'un fichier ajouté par `ADD` ou `COPY`) entraîne une reconstruction des images à partir de la couche affectée.
- Chaque construction d'un Dockerfile génère une **nouvelle** image (modulo une utilisation totale du cache).
- Chaque instruction est **indépendante** : cela n'a pas de sens d'écrire `RUN cd /tmp` —

```console
Step 2/4 : RUN cd /tmp
 ---> Running in d172a17e8705
 ---> 1eb259884a9f
Step 3/4 : RUN pwd
 ---> Running in 80a06bfd0434
/
 ---> 9f59028b0464
```

- Pour changer de répertoire pour les instructions suivantes (y compris pour d'éventuelles images enfantes), utilisez l'instruction `WORKDIR`.

<Exercise id="cmd-vs-entrypoint" level={2}>
  Différence entre les instructions `CMD` et `ENTRYPOINT` : construisez l'image du Dockerfile suivant avec `docker build . -t ping` :

  ```dockerfile
  FROM ubuntu:24.04
  RUN DEBIAN_FRONTEND=noninteractive apt update && \
      apt install -yq iputils-ping && \
      apt clean
  ENTRYPOINT ["ping", "-c", "2"]
  CMD ["localhost"]
  ```

  - Instanciez cette image sans surcharge avec `docker run ping`.
  - Instanciez cette image en surchargeant avec `docker run ping qwant.com`.

  Que constatez-vous ?

  <Fragment slot="solution">
    - `ENTRYPOINT` permet de se servir d'un conteneur comme d'un **exécutable** : les arguments passés à `docker run` ne remplacent pas l'exécutable, ils remplacent `CMD`.
    - On peut surcharger `ENTRYPOINT` avec l'option `--entrypoint`.
    - On peut modifier les paramètres de l'exécutable défini dans `ENTRYPOINT` en surchargeant `CMD` (ici, `localhost` est remplacé par `qwant.com` : le conteneur exécute `ping -c 2 qwant.com`).
    - On peut figer des paramètres dans `ENTRYPOINT` (ici `-c 2`), qui ne seront pas surchargés par `CMD`.
  </Fragment>
</Exercise>
