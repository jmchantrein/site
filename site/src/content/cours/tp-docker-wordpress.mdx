## Introduction

<Note type="warning" title="Aux lecteurs avertis">
  La manière dont est traitée notre première section ne correspond pas à une bonne méthode de travail. Cette section présente néanmoins l'avantage de bien comprendre certains mécanismes de Docker et les différents usages possibles et problématiques rencontrées. Cette section nous permettra également de comprendre pourquoi, lorsque l'on utilise Docker, il vaut mieux réaliser une architecture de microservices plutôt qu'une architecture monolithique.
</Note>

Tout d'abord, la section **Dockerfile** vise à nous emmener étape par étape vers l'utilisation de docker-compose. Nous installerons tout un service WordPress dans une seule image. Cela nous permettra de prendre en main la rédaction d'un Dockerfile. Nous nous interrogerons au fur et à mesure sur la mise en place de bonnes pratiques à la rédaction de Dockerfile. Nous pointerons du doigt les problèmes qui sont engendrés lorsque plusieurs processus sont exécutés dans un seul conteneur et nous proposerons une architecture de microservices permettant de pallier ces problèmes. Cet aboutissement se fera étape par étape. **Il est tout à fait inutile de vouloir griller une étape, chaque étape supérieure repose sur la compréhension et la réalisation des étapes inférieures.**

Ensuite, dans la section **Docker Compose**, nous verrons les méthodes de travail plus adaptées et plus simples pour mettre en place un service Docker en production sur un serveur. En l'occurrence, il s'agira toujours d'un service WordPress.

Enfin, dans la section **Docker Swarm**, nous verrons comment fonctionne Docker Swarm de manière générale, nous présenterons brièvement les fonctionnalités de répartition de charge et de mise à l'échelle et enfin nous orchestrerons notre architecture de microservices WordPress dans un cluster de machines Docker.

## Dockerfile : du Dockerfile au fichier compose par l'exemple

### Étape 1 : création basique d'un Dockerfile

<Note title="Conseil de méthode">
  Lors de la réalisation de cette étape, n'hésitez pas à user et à abuser de la notion de contexte que nous fournit `docker build` (notion de cache). Pour cela, pensez bien à rédiger les instructions les plus coûteuses en début de Dockerfile (i.e. les couches les plus basses), et les instructions les plus souvent amenées à être changées en fin de Dockerfile (i.e. les couches les plus hautes). De cette manière, tout changement dans les couches hautes n'affectera pas les couches basses, qui seront réutilisées par le cache de `docker build`. De plus, vous ne chercherez pas à minimiser le nombre d'images intermédiaires pour l'instant, nous le ferons lors de la prochaine étape.
</Note>

Vous travaillerez dans un répertoire nommé `1_wordpress_multiservice_dirty`.

Pensez à tester votre conteneur à l'adresse [http://localhost:80/wordpress](http://localhost:80/wordpress).

#### Quelques contraintes

- L'image de base est l'image officielle `debian:latest`.
- Il est interdit d'installer le paquet `wordpress`.
- Il faut installer les paquets nécessaires à l'utilisation de WordPress (on remplacera `mysql-server` par `mariadb-server`).
- Il faut télécharger les [sources](https://wordpress.org/latest.zip) au format zip.
- Les sources seront placées dans le répertoire `/var/www/html`.
- Le mot de passe, le nom de l'administrateur MySQL et le nom de la base de données doivent être définis dans des variables d'environnement (mot-clé `ENV` en une seule instruction).
- L'utilisateur MySQL aura l'identifiant `admin`, le mot de passe `tempo` et tous les droits sur une base `wordpress`.
- La commande par défaut doit exécuter Apache et MySQL.
- L'image sera taguée `mon_image_wordpress` et le conteneur exécuté sera nommé `mon_conteneur_wordpress`.
- La fin du fichier Dockerfile contiendra en commentaires les instructions de construction et d'instanciation de l'image.

#### Quelques astuces, pistes de réflexion

- MariaDB est un fork de MySQL : les commandes historiques de MySQL fonctionnent encore et les exécutables `mysql` sont en fait des liens symboliques vers des exécutables MariaDB.
- `apt-get install -s` ou `apt install -s` permet de simuler l'installation d'un paquet.
- `apt depends` permet de savoir quelles sont les dépendances d'un paquet.
- Contrairement à ce que l'on peut voir partout, y compris dans mon cours, il ne faut pas déclarer une variable d'environnement `DEBIAN_FRONTEND=noninteractive` mais plutôt ([voir le ticket GitHub](https://github.com/moby/moby/issues/4032)) :

```bash
RUN DEBIAN_FRONTEND=noninteractive apt install package1 package2 ...
```

- Si vous n'avez besoin de changer de chemin que pour une ou des commandes, n'utilisez pas le mot-clé `WORKDIR` (sa portée s'étend à toutes les instructions suivantes, y compris la commande par défaut du conteneur). Préférez l'utilisation de :

```bash
RUN cd /mon/chemin && mes_commandes
```

- Création d'un utilisateur MySQL `username` avec mot de passe `userpassword` ayant tous les droits, et création d'une base `userbase` :

```bash
mysql -u root --execute="CREATE USER 'username'@'%' IDENTIFIED BY 'userpassword'"
mysql -u root --execute="GRANT ALL PRIVILEGES ON *.* TO 'username'@'%' WITH GRANT OPTION"
mysql -u root --execute="FLUSH PRIVILEGES"
mysql --user=username --password=userpassword --execute="CREATE DATABASE userbase"
```

<Note title="N.B.">
  Pour que les commandes ci-dessus fonctionnent, il ne faut pas oublier de démarrer le serveur MariaDB. Pour cela, il faut s'assurer que le répertoire `/run/mysqld` existe bien dans le conteneur et que `mysql` est bien le propriétaire de `/run/mysqld`.
</Note>

- N'oubliez pas que les instructions sont effectuées par l'utilisateur `root` par défaut. Cela pourrait poser des problèmes de droits par la suite.
- Vous allez exécuter un conteneur contenant plusieurs services : je vous propose de lire cet [article](https://docs.docker.com/engine/containers/multi-service_container/).
- Lorsque vous allez exécuter votre conteneur, il va falloir que celui-ci communique depuis son réseau (par défaut, le réseau *bridge* qui « NAT » le trafic des conteneurs vers l'extérieur) vers le réseau de l'hôte. Vous allez donc devoir mapper/associer un port de l'hôte avec un port du conteneur. Cela se fait via l'option `-p` de `docker run`. Par exemple, la commande suivante permet de mapper le port 8000 de l'hôte au port 80 du conteneur de l'image nginx :

```bash
docker run -p 8000:80 nginx
# Vous pourrez observer les logs ici lorsque vous vous connecterez à l'adresse http://localhost:8000
```

#### Quelques questions, remarques

Ici, vous exécutez plusieurs processus dans un conteneur. La philosophie de Docker veut que l'on ne conteneurise qu'un seul processus. C'est un choix discutable, mais, si l'on souhaite vraiment utiliser plus d'un processus dans un conteneur en production, il faudrait alors se servir de [supervisord](http://supervisord.org/). Nous ne nous en servirons pas dans le cadre de ce cours.

<Exercise id="mode-detache" level={2}>
  Comment fait-on pour lancer le conteneur en mode détaché (`-d` à la place de `-it`) ?

  Avant de consulter la solution, essayez les commandes suivantes et observez ce qui se passe :

  ```bash
  docker run -it --rm --hostname test --name test ubuntu /bin/bash
  docker run -it --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity"
  docker run -it --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity & ps -eflH"
  docker run -d --rm --hostname test --name test ubuntu /bin/bash
  ```

  Pourquoi le conteneur de la dernière commande ne reste-t-il pas actif ?

  <Fragment slot="solution">
    On instancie un conteneur Ubuntu en mode interactif et on exécute la commande `/bin/bash` :

    ```console
    jmc@laptop ~ $ docker run -it --rm --hostname test --name test ubuntu /bin/bash
    root@test:/# exit
    jmc@laptop ~ $
    ```

    On a le résultat attendu. On instancie maintenant un conteneur Ubuntu en mode interactif et on exécute la commande `/bin/bash -c "sleep infinity"` :

    ```console
    jmc@laptop ~ $ docker run -it --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity"
    ```

    On a le comportement attendu : le conteneur ne rend pas la main puisque la commande `sleep infinity` est en mode *foreground*. Depuis un autre terminal, faire :

    ```bash
    docker container rm -f test
    ```

    On instancie un conteneur Ubuntu en mode interactif et on exécute la commande `/bin/bash -c "sleep infinity & ps -eflH"` :

    ```console
    jmc@laptop ~ $ docker run -it --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity & ps -eflH"
    F S UID        PID  PPID  C PRI  NI ADDR SZ WCHAN  STIME TTY          TIME CMD
    4 S root         1     0  0  80   0 -  4507 wait   15:51 pts/0    00:00:00 /bin/
    0 S root         7     1  0  80   0 -  1096 hrtime 15:51 pts/0    00:00:00   sle
    0 R root         8     1  0  80   0 -  8607 -      15:51 pts/0    00:00:00   ps
    jmc@laptop ~ $
    ```

    Le conteneur s'arrête alors même que l'on souhaiterait qu'il exécute la commande `sleep infinity`. Cela s'explique par le fait que la commande `sleep infinity` est exécutée en tâche de fond (utilisation de `&`). Puis la commande `ps -eflH` s'exécute… et il n'y a plus de tâche en mode *foreground*. Or les conteneurs sont conçus pour qu'il y ait une [tâche en mode *foreground* qui soit exécutée](https://docs.docker.com/engine/containers/run/#foreground-and-background).

    Lorsque vous exécutez un conteneur en mode détaché (`--detach`, `-d`), il n'y a pas d'interaction avec le conteneur : `/bin/bash` ne peut donc pas rester en mode *foreground* :

    ```console
    jmc@laptop ~ $ docker run -d --rm --hostname test --name test ubuntu /bin/bash
    6e15d2da91b77de6dea905f00b14877352d50e0fdd3e747e99147a923037e63d
    jmc@laptop ~ $ docker container ls -a
    CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS              PORTS               NAMES
    ```

    `/bin/bash`, n'ayant pas d'interaction, est passé en mode *background* ; le conteneur `test` est devenu inactif et il a été supprimé puisque l'option `--rm` a été activée. Pour conserver le conteneur actif, il faut absolument une tâche en mode *foreground* dans le conteneur. Or c'est rarement le cas des démons de serveurs (apache2, mysql, nginx…) à qui on demande d'habitude de justement fonctionner en mode *background* ! De fait, il faut trouver les options nécessaires à l'utilisation de nos serveurs en mode *foreground* (par exemple : `nginx -g 'daemon off;'`).

    Quid de l'exécution de plusieurs serveurs dans un conteneur ? Il faut alors des serveurs en mode *background* et un serveur en mode *foreground*, ce qui n'est ni homogène, ni pratique. De plus, la commande `docker logs` ne nous fournira que les logs du serveur en mode *foreground* et pas les logs des serveurs en mode *background*.

    Dernier point : on peut obtenir de l'interaction avec un conteneur en mode détaché de la manière suivante :

    ```console
    jmc@laptop ~ $ docker run -d --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity"
    c08701adde66f6316399989fd37d83e71acec35e7564444f1afbdf675a76c08b
    jmc@laptop ~ $ docker container exec -it test /bin/bash
    root@test:/# exit
    jmc@laptop ~ $ docker container stop test
    ```
  </Fragment>
</Exercise>

### Étape 3 : le problème de l'utilisateur root

Tout d'abord, un peu de [lecture](https://docs.docker.com/engine/security/) sur la sécurité lorsque l'on utilise un environnement Docker, ou bien directement la [conclusion](https://docs.docker.com/engine/security/#conclusions).

<Note type="info" title="Évolution · exécuter sans root">
  - **Avant.** Pour qu'un serveur lié au port 80 tourne sans être `root`, on posait un bit *setuid* sur le binaire — exactement ce que fait ce TP, à dessein pédagogique.
  - **Le problème.** Le *setuid* est une rustine fragile : si le binaire est vulnérable, c'est une porte ouverte à l'élévation de privilèges ; et le conteneur démarre malgré tout en `root`.
  - **Aujourd'hui.** Une image déclare un `USER` non privilégié ; Docker propose le **mode rootless** (le démon lui-même tourne sans `root`) et le réglage fin des *capabilities* (`--cap-drop ALL --cap-add NET_BIND_SERVICE`). Les **images officielles** abandonnent proprement les privilèges via un *entrypoint* + `gosu`/`su-exec`.
  - **Pourquoi c'est mieux.** Surface d'attaque réduite, plus de `setuid` artisanal, conforme aux bonnes pratiques. C'est d'ailleurs pourquoi les images officielles des étapes 7-8 « fonctionnent toutes seules », sans ces contorsions.
</Note>

### Étape 4 : pérennisation des données

<Note type="info" title="Évolution · les drivers de stockage">
  - **Avant.** Docker proposait plusieurs *storage drivers* pour empiler les couches : `aufs`, `devicemapper`, `btrfs`, `overlay`… à choisir selon le noyau de l'hôte.
  - **Le problème.** `aufs` n'a jamais été intégré au noyau Linux mainline ; `devicemapper` en mode *loopback* était lent et réputé fragile en production.
  - **Aujourd'hui.** **overlay2** est le driver par défaut et recommandé partout ; `aufs` et `devicemapper` sont dépréciés et retirés (Docker Engine 25+). Dans l'immense majorité des cas, on ne choisit plus rien.
  - **Pourquoi c'est mieux.** overlay2 est dans le noyau mainline, rapide et stable. À ne pas confondre avec les **volumes** (l'objet de cette étape), qui gèrent la *persistance* des données, là où le driver gère l'*empilement des couches* d'images.
</Note>

#### Quelques contraintes

- Copiez votre répertoire `3_wordpress_multiservice_noroot` en `4_wordpress_multiservice_persistant`.
- Modifiez votre Dockerfile et/ou votre commande de lancement de conteneur, de manière à pouvoir faire persister les données présentes dans `/var/lib/mysql` et `/var/www/html`.
- Modifiez explicitement votre Dockerfile pour exposer le port 80 des futurs conteneurs.

#### Quelques astuces, pistes de réflexion

- Nous utiliserons ici l'instruction `VOLUME` dans notre Dockerfile.

#### Méthodes pour la pérennisation des données

1. Nous avons trouvé une méthode pour pérenniser nos données dans un volume. Lorsque ce volume est dans l'arborescence du système de fichiers de Docker (Docker a son propre [backend de stockage (overlay2 par défaut aujourd'hui)](https://docs.docker.com/engine/storage/drivers/select-storage-driver/)), les données initialement contenues dans l'image sont copiées lors du premier lancement dans le volume. Ensuite les données sont ajoutées dans le volume.

2. Nous pourrions faire un point de montage de `/var/lib/mysql` dans notre propre système de fichiers (par exemple : `/data`). Cependant, lors du premier montage, toutes les données contenues dans `/var/lib/mysql` de l'image sont inaccessibles, car le dossier `/data` monté est initialement vide : le conteneur serait inutilisable car MySQL n'aurait même plus les informations concernant ses propres utilisateurs. On peut contourner ce problème en lançant un script au démarrage du conteneur, qui regarde si le point de montage `/data` est vide ou non. S'il est vide, on initialise MySQL, sinon on ne fait rien. L'initialisation se fait donc une seule fois lors du premier démarrage du conteneur et elle ne se fait pas dans l'image. Vous pouvez voir cet [exemple](https://web.archive.org/web/2017/http://txt.fliglio.com/2013/11/creating-a-mysql-docker-container/) (archive d'un article de 2013). Peut-être trouvez-vous cela bien compliqué et contraignant ? Cependant, un énorme avantage de cette méthode, en plus de localiser ses données où on le souhaite, c'est la possibilité de passer ses propres variables d'environnement à l'instanciation du conteneur : par exemple, on choisit quel est le mot de passe utilisateur au lancement du conteneur et non de manière figée dans l'image. Cette méthode nous apporte donc beaucoup plus de souplesse dans la gestion de nos conteneurs. On ne mettra pas en œuvre cette méthode dans cette partie. Mais gardez à l'esprit que ce type de mécanisme peut être inclus dans des images dont vous pourriez vous servir. Ce sera d'ailleurs le cas du microservice officiel MySQL dont nous nous servirons dans la prochaine partie.

   On pourrait remarquer aussi que la mise en place d'une telle complexité ne peut être envisageable que pour une instanciation massive de conteneurs WordPress et non pas pour quelques instances. C'est vrai, mais gardez à l'esprit que nous sommes dans le cadre d'un TP de compréhension des mécanismes de Docker : les images officielles implémentent elles-mêmes ce type de mécanisme, et de manière bien plus efficace que ce que nous faisons ici. Ainsi, un microservice devrait pouvoir être déployé tout aussi bien sporadiquement que massivement.

3. Une autre alternative consisterait à faire un point de montage ad hoc. On monte un répertoire `/backup` de l'hôte sur un répertoire `/backup` dans le conteneur. Un cron sur l'hôte réalise régulièrement des dumps de la base de données via une commande du style :

   ```bash
   docker exec -it my_wordpress_container mysqldump --all-databases > /backup/$(date)_dump.sql
   ```

   Notez que le cron se ferait sur l'hôte, et pas dans le conteneur : minimisation du nombre de processus dans le conteneur.

4. Nous pourrions aussi pérenniser les sources (`/var/www/html`) avec un montage sur l'hôte. Dans le cas d'un montage hors du système de fichiers de Docker (cf. notre solution numéro 2), cela nécessiterait l'écriture d'un bootstrap sur lequel on mettrait un bit setuid ; le bootstrap modifierait le propriétaire du répertoire monté par `www-data:www-data` avec `chown` ; il faut donc être `root` pour cette exécution (d'où l'usage du setuid).

**Conclusion.** La méthode 1 est la plus facile à mettre en œuvre. C'est la méthode officielle recommandée par Docker, mais elle nous ôte la liberté de choisir où nous stockons nos données. La méthode 2 est la plus difficile à mettre en œuvre mais offre plus de souplesse dans la gestion de nos services ; cependant, contrairement à la méthode 1, nous ne pouvons pas gérer les données via le CLI de Docker (`docker volume cmd`). Les méthodes 3 et 4 sont, à mon avis, à proscrire. Vous trouverez plus d'informations sur la méthode 1 [ici](https://docs.docker.com/engine/storage/volumes/) et sur la méthode 2 [ici](https://docs.docker.com/engine/storage/bind-mounts/).

<Note type="warning" title="Nommez vos volumes">
  Lorsque vous utilisez la méthode 1, vous devez [nommer vos volumes](https://docs.docker.com/engine/storage/volumes/). En effet, les volumes anonymes sont supprimés à la suppression du conteneur (option `--rm`) ou lors de l'utilisation de la commande `docker volume prune`.
</Note>

<Note type="danger" title="Attention à docker volume prune">
  Historiquement, `docker volume prune` supprimait **aussi** les volumes nommés qui n'étaient utilisés par aucun conteneur. Depuis Docker 23, la commande ne supprime par défaut que les volumes **anonymes** ; les volumes nommés inutilisés ne sont supprimés qu'avec `docker volume prune --all`. Dans tous les cas, un volume nommé inutilisé n'est pas à l'abri d'un prune : nommer ne suffit pas, sauvegardez.
</Note>

<Note type="info" title="Évolution · docker volume prune">
  - **Avant.** `docker volume prune` supprimait **tous** les volumes inutilisés, y compris les volumes **nommés** — un simple nettoyage pouvait effacer des données qu'on croyait protégées par leur nom.
  - **Le problème.** Comportement trop agressif et contre-intuitif : nommer un volume ne suffisait pas à le mettre à l'abri d'un `prune`.
  - **Aujourd'hui.** Depuis **Docker 23** (2023), `docker volume prune` ne retire par défaut que les volumes **anonymes** ; il faut `--all` pour inclure les volumes nommés inutilisés.
  - **Pourquoi c'est mieux.** Le comportement par défaut est désormais sûr (les volumes nommés survivent à un nettoyage courant) — sans pour autant dispenser de **sauvegarder**, comme le rappelle l'avertissement ci-dessus.
</Note>

  Si vous devez gérer des données temporaires, il vaut mieux utiliser la directive [`--tmpfs`](https://docs.docker.com/engine/storage/tmpfs/) plutôt que d'utiliser la couche en écriture de votre conteneur.
</Note>

<Note type="info" title="Évolution · de --link aux réseaux définis par l'utilisateur">
  - **Avant.** Pour qu'un conteneur en joigne un autre par son nom, on les reliait avec `--link` (ce que fait cette étape, à dessein).
  - **Le problème.** `--link` est **déprécié** : il ne fonctionne que sur le réseau *bridge* par défaut, exige un ordre de démarrage précis et n'offre pas de vraie résolution dynamique.
  - **Aujourd'hui.** On crée un **réseau défini par l'utilisateur** (`docker network create`) : Docker y fournit une **résolution DNS automatique** par nom de conteneur. Docker Compose (étape suivante) crée ce réseau tout seul — d'où la disparition de `--link`.
  - **Pourquoi c'est mieux.** Plus d'ordre de démarrage imposé, isolation par service, et un nom de service qui résout de lui-même vers la bonne adresse.
</Note>

#### Quelques questions, remarques

- En production, on évitera l'usage de l'option `--link` qui est certainement vouée à disparaître. Idéalement, il faut créer un sous-réseau dédié pour chacun des services que l'on met en place avec Docker. Ceci peut se faire via la commande `docker network`, mais nous verrons plus tard comment automatiser ce processus.
- Notez que l'on n'a pas exposé le port du conteneur MySQL à l'extérieur : seul le conteneur Apache y a accès.
- Nous sommes en présence d'une architecture de 2 microservices : que pensez-vous du mécanisme de script `build_and_launch.sh` ?
- Et si nous étions en présence d'une architecture de 25 microservices, que penseriez-vous du mécanisme de script `build_and_launch.sh` ?
- Docker va plus loin que la simple rédaction de Dockerfile : Docker nous permet aussi de gérer la façon dont interagissent les conteneurs entre eux. Pour cela, Docker met à notre disposition l'outil Docker Compose. C'est l'objet de notre prochaine partie.

## Docker Compose, ou l'art d'orchestrer nos conteneurs sur une seule machine

<Note title="Installation de Docker Compose">
  Docker Compose (v2) est aujourd'hui un **plugin** du client Docker, fourni par le paquet `docker-compose-plugin` du dépôt officiel ([documentation](https://docs.docker.com/compose/install/linux/)) :

  ```bash
  sudo apt update && sudo apt install docker-compose-plugin
  docker compose version
  ```

  La commande s'écrit `docker compose` (avec une espace). L'ancien binaire autonome `docker-compose` (v1), que ce TP installait par curl à l'époque, est en fin de vie depuis 2023.
</Note>

<Note type="info" title="Évolution · Docker Compose (v1 → v2)">
  - **Avant.** Compose était un **binaire autonome** `docker-compose` (en Python), installé à part (souvent par `curl`), et les fichiers commençaient par une clé `version:`.
  - **Le problème.** Outil séparé du client Docker, plus lent et désynchronisé de l'évolution du moteur ; la clé `version:` prêtait à confusion.
  - **Aujourd'hui.** Compose v2 est un **plugin** du client Docker (en Go) : on écrit `docker compose` (avec une espace), la clé `version:` est **obsolète**, et le binaire v1 est en fin de vie depuis 2023.
  - **Pourquoi c'est mieux.** Un seul outil intégré et maintenu, cohérent avec le reste du CLI. Sur les vieux tutoriels, méfiez-vous de `docker-compose` (avec un tiret) et de la clé `version:`.
</Note>

<Note type="info" title="depends_on n'attend pas que le service soit prêt">
  Attention à un piège classique : `depends_on` garantit l'**ordre de démarrage**, pas la **disponibilité**. Compose démarre bien `mysql_wordpress` avant `apache_wordpress`, mais n'attend pas que MariaDB **accepte les connexions** ; au premier lancement, l'installateur WordPress peut donc se présenter avant que la base soit prête. La parade moderne : déclarer un `HEALTHCHECK` sur la base et **conditionner** le démarrage —

  ```yaml
  services:
    mysql_wordpress:
      healthcheck:
        test: ["CMD", "mysqladmin", "ping", "-h", "127.0.0.1", "--silent"]
        interval: 5s
        retries: 10
    apache_wordpress:
      depends_on:
        mysql_wordpress:
          condition: service_healthy
  ```

  (Notre corrigé en reste au `depends_on` simple, suffisant ici parce que l'installation de WordPress est manuelle et postérieure.)
</Note>

## Docker Swarm, ou l'art d'orchestrer nos conteneurs sur un cluster de machines Docker

<Note title="Prérequis">
  Il vous faut un [compte Docker](https://hub.docker.com/) pour pouvoir réaliser cette partie.
</Note>

### Introduction

Docker Swarm est un orchestrateur. C'est un outil permettant de gérer vos images, conteneurs et réseaux virtuels sur un ensemble de machines utilisant Docker (cluster ou grappe). Voici les principales fonctionnalités de Docker Swarm :

- Gestion d'un cluster de démons Docker.
- Décentralisation d'exécution (un conteneur `a` sur une machine A peut communiquer avec un conteneur `b` sur une machine B).
- Utilisation d'un fichier au format compose pour le déploiement (modèle déclaratif).
- Passage à l'échelle : pour chaque service, vous pouvez déclarer le nombre de conteneurs que vous souhaitez exécuter. Lorsque vous augmentez ou réduisez ce nombre de conteneurs, Docker Swarm s'adapte automatiquement en ajoutant ou en supprimant des conteneurs pour maintenir l'état souhaité. Cela nécessite toutefois d'avoir pensé une architecture de microservices pour son service.
- Adaptation de l'état général du cluster : si un des nœuds du cluster tombe, Swarm redéploie automatiquement les conteneurs perdus sur les nœuds restants et actifs.
- Réseaux virtuels partagés entre plusieurs machines (*overlay network*).
- Équilibrage de charge : chaque service de l'essaim a un nom DNS unique et Docker Swarm répartit la charge entre les conteneurs en cours d'exécution. Vous pouvez interroger chaque conteneur qui s'exécute dans le cluster via un serveur DNS intégré. Vous pouvez aussi exposer les ports des services à un équilibreur de charge externe. Mais en interne, Docker Swarm permet de spécifier comment distribuer les conteneurs d'un service entre les nœuds du cluster.
- Communication entre les nœuds sécurisée par défaut via TLS.
- Mise à jour en cours d'exécution et retour arrière : lorsque vous devez faire une mise à jour d'un service, vous pouvez appliquer ces mises à jour de service aux nœuds de manière incrémentielle. Si quelque chose ne va pas, vous pouvez restaurer un service vers une version précédente.

<Note title="Vocabulaire">
  Dans Swarm, nous allons introduire un nouveau terme : la pile (*stack*). La pile correspond à ce que nous appelons le service dans sa globalité (dans notre précédent exemple, on parlerait de pile WordPress) ; le service correspond à ce que nous appelions le microservice ; les conteneurs, quant à eux, sont appelés des tâches (*tasks*). Un microservice répliqué n fois correspond à l'instanciation de n conteneurs.
</Note>

<Note type="info" title="Évolution · de Swarm à Kubernetes">
  - **Avant.** Pour orchestrer des conteneurs sur plusieurs machines, **Docker Swarm** — intégré au moteur Docker — était le choix simple et naturel.
  - **Le problème.** Swarm reste facile à prendre en main mais offre moins de fonctionnalités, et son développement s'est essentiellement arrêté.
  - **Aujourd'hui.** **Kubernetes** (CNCF) s'est imposé comme le **standard de fait** de l'orchestration : plus mature, immense écosystème, adopté massivement par l'industrie. Swarm, toujours fonctionnel, n'évolue plus guère.
  - **Pourquoi c'est mieux.** Portabilité, communauté et fonctionnalités. Swarm garde toutefois sa valeur ici : un excellent **marchepied conceptuel** — ses notions (pile, service, tâche) se transposent à Kubernetes.
</Note>

Vous pouvez modifier le nombre de réplicas en mettant à jour votre (micro)service :

```bash
docker service update --replicas 2 voting_stack_vote

```

En fonction de ce qui a été défini dans le fichier compose et des éventuelles mises à jour disponibles, vous pouvez mettre à jour la pile (le service dans sa globalité) de cette manière :

```bash
docker stack deploy --compose-file docker-stack.yml
```

Vous pourriez essayer en modifiant le mapping du port 5000 en 6000 si vous souhaitez voir un changement. Avec la commande :

```console
docker service ps voting_stack_vote
```

<Note type="danger" title="Montages liés et cluster multi-nœuds">
  L'étape 8 persiste les données via des **montages liés** (*bind mounts*) vers `$HOME` (`source_code_wordpress`, `data_wordpress`). Or un *bind mount* est **local à un nœud** : sur un Swarm de plusieurs machines, le code et la base n'existent que sur le nœud où vous les avez créés, et **répartir Apache sur plusieurs nœuds** casse la cohérence (chaque réplica verrait un disque différent, voire vide). Pour un vrai cluster, il faut soit **contraindre le placement** des services à état sur un nœud précis (`deploy.placement.constraints`), soit un **stockage partagé** (volume NFS/cluster, ou une base managée). Sur play-with-docker, restez donc sur une démonstration **mono-nœud** pour les services à état.
</Note>
