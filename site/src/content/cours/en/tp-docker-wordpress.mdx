## Introduction

<Note type="warning" title="To experienced readers">
  The way our first section is handled does not reflect good working practice. This section nevertheless has the advantage of giving a solid understanding of certain Docker mechanisms, of the various possible uses, and of the problems encountered. It will also help us understand why, when using Docker, it is better to build a microservices architecture rather than a monolithic one.
</Note>

First of all, the **Dockerfile** section aims to lead us, step by step, towards the use of docker-compose. We will install an entire WordPress service in a single image. This will let us get to grips with writing a Dockerfile. As we go along, we will reflect on establishing good practices for writing Dockerfiles. We will point out the problems that arise when several processes run inside a single container, and we will propose a microservices architecture that overcomes these problems. We will get there step by step. **It is completely pointless to try to skip a step: each higher step builds on the understanding and completion of the steps below it.**

Next, in the **Docker Compose** section, we will look at working methods that are better suited and simpler for deploying a Docker service into production on a server. In this case, it will still be a WordPress service.

Finally, in the **Docker Swarm** section, we will see how Docker Swarm works in general, briefly present its load-balancing and scaling features, and then orchestrate our WordPress microservices architecture in a cluster of Docker machines.

## Dockerfile: from the Dockerfile to the compose file, by example

### Step 1: basic creation of a Dockerfile

<Note title="Methodology tip">
  While working through this step, feel free to make heavy use of the context notion that `docker build` provides (the cache mechanism). To do so, remember to put the most expensive instructions at the beginning of the Dockerfile (i.e. the lowest layers), and the instructions most likely to change at the end of the Dockerfile (i.e. the highest layers). That way, any change in the upper layers will not affect the lower layers, which will be reused by the `docker build` cache. Also, do not try to minimize the number of intermediate images for now; we will do that in the next step.
</Note>

You will work in a directory named `1_wordpress_multiservice_dirty`.

Remember to test your container at [http://localhost:80/wordpress](http://localhost:80/wordpress).

- The base image is the official `debian:latest` image.
- Installing the `wordpress` package is forbidden.

- You must download the [sources](https://wordpress.org/latest.zip) in zip format.

- The password, the MySQL administrator name and the database name must be defined in environment variables (`ENV` keyword in a single instruction).
- The MySQL user will have the username `admin`, the password `tempo` and full privileges on a `wordpress` database.
- The default command must run Apache and MySQL.
- The image will be tagged `mon_image_wordpress` and the running container will be named `mon_conteneur_wordpress`.
- The end of the Dockerfile must contain, as comments, the instructions for building and instantiating the image.

#### A few tips and food for thought

- MariaDB is a fork of MySQL: the historical MySQL commands still work, and the `mysql` executables are actually symbolic links to MariaDB executables.
- `apt-get install -s` or `apt install -s` lets you simulate the installation of a package.
- `apt depends` tells you what the dependencies of a package are.
- Contrary to what you can see everywhere, including in my course, you should not declare a `DEBIAN_FRONTEND=noninteractive` environment variable but rather ([see the GitHub issue](https://github.com/moby/moby/issues/4032)):

```bash
RUN DEBIAN_FRONTEND=noninteractive apt install package1 package2 ...
```

- If you only need to change directory for one or a few commands, do not use the `WORKDIR` keyword (its scope extends to all subsequent instructions, including the container's default command). Prefer:

```bash
RUN cd /mon/chemin && mes_commandes
```

- Creating a MySQL user `username` with password `userpassword` having full privileges, and creating a `userbase` database:

```bash
mysql -u root --execute="CREATE USER 'username'@'%' IDENTIFIED BY 'userpassword'"
mysql -u root --execute="GRANT ALL PRIVILEGES ON *.* TO 'username'@'%' WITH GRANT OPTION"
mysql -u root --execute="FLUSH PRIVILEGES"
mysql --user=username --password=userpassword --execute="CREATE DATABASE userbase"
```

<Note title="N.B.">
  For the commands above to work, do not forget to start the MariaDB server. To do so, make sure the `/run/mysqld` directory actually exists in the container and that `mysql` is indeed the owner of `/run/mysqld`.
</Note>

- Do not forget that instructions are executed by the `root` user by default. This could cause permission problems later on.
- You are going to run a container hosting several services: I suggest you read this [article](https://docs.docker.com/engine/containers/multi-service_container/).
- When you run your container, it will have to communicate from its own network (by default, the *bridge* network, which "NATs" container traffic towards the outside) to the host's network. You will therefore have to map/bind a host port to a container port. This is done via the `-p` option of `docker run`. For example, the following command maps port 8000 of the host to port 80 of the nginx image's container:

```bash
docker run -p 8000:80 nginx
# You will be able to watch the logs here when you connect to http://localhost:8000
```

#### A few questions and remarks

Here, you are running several processes in one container. Docker's philosophy is to containerize only one process. This is a debatable choice, but if you really want to run more than one process in a container in production, you should then use [supervisord](http://supervisord.org/). We will not use it in this course.

<Exercise id="mode-detache" level={2}>
  How do you launch the container in detached mode (`-d` instead of `-it`)?

  Before looking at the solution, try the following commands and observe what happens:

  ```bash
  docker run -it --rm --hostname test --name test ubuntu /bin/bash
  docker run -it --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity"
  docker run -it --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity & ps -eflH"
  docker run -d --rm --hostname test --name test ubuntu /bin/bash
  ```

  Why does the container from the last command not stay running?

  <Fragment slot="solution">
    We instantiate an Ubuntu container in interactive mode and run the `/bin/bash` command:

    ```console
    jmc@laptop ~ $ docker run -it --rm --hostname test --name test ubuntu /bin/bash
    root@test:/# exit
    jmc@laptop ~ $
    ```

    We get the expected result. We now instantiate an Ubuntu container in interactive mode and run the command `/bin/bash -c "sleep infinity"`:

    ```console
    jmc@laptop ~ $ docker run -it --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity"
    ```

    We get the expected behavior: the container does not give control back, since the `sleep infinity` command runs in *foreground* mode. From another terminal, run:

    ```bash
    docker container rm -f test
    ```

    We instantiate an Ubuntu container in interactive mode and run the command `/bin/bash -c "sleep infinity & ps -eflH"`:

    ```console
    jmc@laptop ~ $ docker run -it --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity & ps -eflH"
    F S UID        PID  PPID  C PRI  NI ADDR SZ WCHAN  STIME TTY          TIME CMD
    4 S root         1     0  0  80   0 -  4507 wait   15:51 pts/0    00:00:00 /bin/
    0 S root         7     1  0  80   0 -  1096 hrtime 15:51 pts/0    00:00:00   sle
    0 R root         8     1  0  80   0 -  8607 -      15:51 pts/0    00:00:00   ps
    jmc@laptop ~ $
    ```

    The container stops even though we would like it to run the `sleep infinity` command. This is explained by the fact that `sleep infinity` is run as a background task (use of `&`). Then the `ps -eflH` command runs... and there is no longer any task in *foreground* mode. Yet containers are designed so that there is a [task running in *foreground* mode](https://docs.docker.com/engine/containers/run/#foreground-and-background).

    When you run a container in detached mode (`--detach`, `-d`), there is no interaction with the container: `/bin/bash` therefore cannot stay in *foreground* mode:

    ```console
    jmc@laptop ~ $ docker run -d --rm --hostname test --name test ubuntu /bin/bash
    6e15d2da91b77de6dea905f00b14877352d50e0fdd3e747e99147a923037e63d
    jmc@laptop ~ $ docker container ls -a
    CONTAINER ID        IMAGE               COMMAND             CREATED             STATUS              PORTS               NAMES
    ```

    `/bin/bash`, having no interaction, switched to *background* mode; the `test` container became inactive and was removed since the `--rm` option was enabled. To keep the container running, there absolutely must be a task in *foreground* mode inside the container. Yet this is rarely the case for server daemons (apache2, mysql, nginx...), which we usually ask to run precisely in *background* mode! So we have to find the options needed to run our servers in *foreground* mode (for example: `nginx -g 'daemon off;'`).

    What about running several servers in one container? You then need servers in *background* mode and one server in *foreground* mode, which is neither consistent nor practical. Moreover, the `docker logs` command will only give us the logs of the server running in *foreground* mode, not the logs of the servers running in *background* mode.

    One last point: you can get interaction with a container in detached mode as follows:

    ```console
    jmc@laptop ~ $ docker run -d --rm --hostname test --name test ubuntu /bin/bash -c "sleep infinity"
    c08701adde66f6316399989fd37d83e71acec35e7564444f1afbdf675a76c08b
    jmc@laptop ~ $ docker container exec -it test /bin/bash
    root@test:/# exit
    jmc@laptop ~ $ docker container stop test
    ```

</Exercise>

### Step 4: making data persistent

#### A few constraints

- Copy your `3_wordpress_multiservice_noroot` directory to `4_wordpress_multiservice_persistant`.
- Modify your Dockerfile and/or your container launch command so that the data in `/var/lib/mysql` and `/var/www/html` persists.
- Explicitly modify your Dockerfile to expose port 80 of future containers.

#### A few tips and food for thought

- Here we will use the `VOLUME` instruction in our Dockerfile.

#### Methods for data persistence

1. We have found a method to persist our data in a volume. When this volume lives inside Docker's filesystem tree (Docker has its own [storage backend (overlay2 by default nowadays)](https://docs.docker.com/engine/storage/drivers/select-storage-driver/)), the data initially contained in the image is copied into the volume on the first launch. After that, data is added to the volume.

2. We could create a mount point for `/var/lib/mysql` in our own filesystem (for example: `/data`). However, on the first mount, all the data contained in the image's `/var/lib/mysql` is inaccessible, because the mounted `/data` folder is initially empty: the container would be unusable since MySQL would no longer even have the information about its own users. We can work around this problem by running a script at container startup that checks whether the `/data` mount point is empty or not. If it is empty, we initialize MySQL; otherwise we do nothing. Initialization therefore happens only once, at the container's first startup, and not in the image. You can look at this [example](https://web.archive.org/web/2017/http://txt.fliglio.com/2013/11/creating-a-mysql-docker-container/) (an archive of an article from 2013). Perhaps you find this rather complicated and constraining? However, one huge advantage of this method, besides storing your data wherever you want, is the ability to pass your own environment variables when instantiating the container: for instance, you choose the user password at container launch rather than having it baked into the image. This method therefore gives us much more flexibility in managing our containers. We will not implement this method in this part. But keep in mind that this kind of mechanism may be included in images you might use. This will in fact be the case for the official MySQL microservice that we will use in the next part.

   One could also point out that putting such complexity in place is only worthwhile for massive instantiation of WordPress containers, not for just a few instances. That is true, but keep in mind that we are in a lab designed to understand Docker's mechanisms: the official images implement this very kind of mechanism themselves, and far more efficiently than what we are doing here. Thus, a microservice should be deployable just as easily sporadically as massively.

3. Another alternative would be an ad hoc mount point. We mount a `/backup` directory of the host onto a `/backup` directory inside the container. A cron job on the host regularly dumps the database via a command along the lines of:

   ```bash
   docker exec -it my_wordpress_container mysqldump --all-databases > /backup/$(date)_dump.sql
   ```

   Note that the cron job would run on the host, not inside the container: minimizing the number of processes in the container.

4. We could also persist the sources (`/var/www/html`) with a mount on the host. In the case of a mount outside Docker's filesystem (cf. our solution number 2), this would require writing a bootstrap script on which we would set a setuid bit; the bootstrap would change the owner of the mounted directory to `www-data:www-data` with `chown`; you therefore need to be `root` for this execution (hence the use of setuid).

**Conclusion.** Method 1 is the easiest to implement. It is the official method recommended by Docker, but it takes away our freedom to choose where we store our data. Method 2 is the hardest to implement but offers more flexibility in managing our services; however, unlike method 1, we cannot manage the data through the Docker CLI (`docker volume cmd`). Methods 3 and 4 are, in my opinion, to be avoided. You will find more information on method 1 [here](https://docs.docker.com/engine/storage/volumes/) and on method 2 [here](https://docs.docker.com/engine/storage/bind-mounts/).

<Note type="warning" title="Name your volumes">
  When using method 1, you must [name your volumes](https://docs.docker.com/engine/storage/volumes/). Indeed, anonymous volumes are deleted when the container is removed (`--rm` option) or when the `docker volume prune` command is used.
</Note>

<Note type="danger" title="Beware of docker volume prune">
  Historically, `docker volume prune` **also** removed named volumes that were not used by any container. Since Docker 23, by default the command only removes **anonymous** volumes; unused named volumes are only removed with `docker volume prune --all`. In any case, an unused named volume is not safe from a prune: naming is not enough — back up your data.
</Note>

<Note title="N.B.">
  If you need to handle temporary data, it is better to use the [`--tmpfs`](https://docs.docker.com/engine/storage/tmpfs/) directive rather than using your container's writable layer.
</Note>

## Docker Swarm, or the art of orchestrating our containers on a cluster of Docker machines

<Note title="Prerequisites">
  You need a [Docker account](https://hub.docker.com/) to be able to complete this part.
</Note>

### Introduction

Docker Swarm is an orchestrator. It is a tool for managing your images, containers and virtual networks across a set of machines running Docker (a cluster). Here are the main features of Docker Swarm:

- Management of a cluster of Docker daemons.
- Decentralized execution (a container `a` on a machine A can communicate with a container `b` on a machine B).
- Use of a compose-format file for deployment (declarative model).
- Scaling: for each service, you can declare the number of containers you want to run. When you increase or decrease this number of containers, Docker Swarm adapts automatically by adding or removing containers to maintain the desired state. This does, however, require having designed a microservices architecture for your service.
- Adaptation of the overall cluster state: if one of the cluster's nodes goes down, Swarm automatically redeploys the lost containers onto the remaining active nodes.
- Virtual networks shared across several machines (*overlay network*).
- Load balancing: each service in the swarm has a unique DNS name and Docker Swarm balances the load among the running containers. You can query every container running in the cluster through a built-in DNS server. You can also expose service ports to an external load balancer. But internally, Docker Swarm lets you specify how to distribute a service's containers among the cluster's nodes.
- Communication between nodes secured by default via TLS.
- Rolling updates and rollback: when you need to update a service, you can apply these service updates to the nodes incrementally. If something goes wrong, you can roll a service back to a previous version.

<Note title="Vocabulary">
  In Swarm, we will introduce a new term: the *stack*. The stack corresponds to what we call the service as a whole (in our previous example, we would speak of the WordPress stack); the service corresponds to what we used to call the microservice; the containers, for their part, are called *tasks*. A microservice replicated n times corresponds to the instantiation of n containers.
</Note>

<Note title="Kubernetes">
  There is a very serious competitor to Swarm: the [Kubernetes](https://kubernetes.io/) orchestrator. It is more mature, more stable and offers more features than Docker Swarm. Furthermore, it has been massively adopted by the industry. Docker Swarm has the advantage of being easier to get started with, but the features presented here are also present in Kubernetes. Since then, the market has decided: Kubernetes has established itself as the orchestration standard and Swarm, while still functional, hardly evolves anymore — it remains an excellent conceptual stepping stone.
</Note>

### Native load-balancing management of a service

- Go to the [play-with-docker](https://labs.play-with-docker.com) website.
- Add 3 instances: these 3 instances correspond to 3 Alpine servers with Docker installed.
- We are now going to create a cluster of Docker servers (a *swarm*):

```bash
docker swarm init --advertise-addr $(hostname -i)
```

- From the output of the command above, copy the `join` command (watch out for line breaks) and paste it into the terminals of the 2 other instances.
- And there is our cluster of Docker servers! Doubtful? Go to your first instance (the *manager*) and run the following command:

```bash
docker node ls
```

- Nodes 2 and 3 are considered *workers*. The command above does not work on workers.
- Go to the first instance and run the following command:

```bash
for ((i=1;i<10;i++)); do docker service create --name service_${i} --replicas 1 alpine ping localhost; done
```

- Now let's observe where these various services are running:

```bash
for ((i=1;i<10;i++)); do docker service ps service_${i}; done
```

#### Initializing Docker Swarm

A stack is a group of (micro)services deployed together. The `docker-stack.yml` file in the current folder will be used to deploy the voting app (the service) as a stack. (Translator's note: take the time to read through the instructions in this file.)

#### Translator's bonus: scaling and updating

```console
gf2bb9vl60mv        voting_stack_vote.5   dockersamples/examplevotingapp_vote:before   node2               Running             Running 42 seconds ago
9csix895fldt        voting_stack_vote.6   dockersamples/examplevotingapp_vote:before   node1               Running             Running 43 seconds ago
```

<Note title="N.B.">
  Had there been more nodes, the load would have been automatically distributed across all the nodes.
</Note>

Now suppose the flow of visitors drops:

```console
docker service scale voting_stack_vote=3
# Let's observe what happens
docker service ps voting_stack_vote
ID                  NAME                  IMAGE                                        NODE                DESIRED STATE       CURRENT STATE
iw5yja1o5s27        voting_stack_vote.4   dockersamples/examplevotingapp_vote:before   node1               Running             Running 54 seconds ago
gf2bb9vl60mv        voting_stack_vote.5   dockersamples/examplevotingapp_vote:before   node2               Running             Running 53 seconds ago
9csix895fldt        voting_stack_vote.6   dockersamples/examplevotingapp_vote:before   node1               Running             Running 54 seconds ago
```

You can change the number of replicas by updating your (micro)service:

```bash
docker service update --replicas 2 voting_stack_vote
# If you ever want to roll back
docker service rollback voting_stack_vote
```

Depending on what has been defined in the compose file and on any available updates, you can update the stack (the service as a whole) like this:

```bash
docker stack deploy --compose-file docker-stack.yml
```

You could try changing the mapping of port 5000 to 6000 if you want to see a change. With the command:

```console
docker service ps voting_stack_vote
ID                  NAME                      IMAGE                                        NODE                DESIRED STATE       CURRENT STATE
xb0xbq451yh5        voting_stack_vote.4       dockersamples/examplevotingapp_vote:before   node2               Running             Running 2 seconds ago
5szo5dbt4zeu         \_ voting_stack_vote.4   dockersamples/examplevotingapp_vote:before   node1               Shutdown            Shutdown 4 seconds ago
tdicx6w98ugk        voting_stack_vote.8       dockersamples/examplevotingapp_vote:before   node1               Running             Running 2 seconds ago
e7l8luq3zfoc         \_ voting_stack_vote.8   dockersamples/examplevotingapp_vote:before   node2               Shutdown            Shutdown 4 seconds ago
```

We notice that this microservice was interrupted for 2 seconds and that there are leftovers. These leftovers can be used to roll back in case of a problem. The service interruption can be avoided by using the `delay` keyword in the docker-compose file, which specifies a delay between the updates of each container: this means that, for a brief moment, there will be 2 different versions of the same microservice online. You will find more information [here](https://docs.docker.com/engine/swarm/swarm-tutorial/rolling-update/).
