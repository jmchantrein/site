## Creating images: the naive method

<Note type="warning" title="Only three use cases">
  - For teaching purposes.
  - When there is no way to configure an environment other than through a graphical interface (Eclipse, for example).
  - To debug an automated build (`docker build`).

### Using docker commit

We modify a container, then freeze its read/write layer into a new image:

```console
jmc@laptop ~ $ docker run -it -h conteneur --name conteneur_avant_commit ubuntu /bin/bash
root@conteneur:/# echo "hello world !" > /nouveau_fichier_dans_le_systeme_de_fichier_racine.txt
root@conteneur:/# exit
```

```console
jmc@laptop ~ $ docker images
```

<Note type="warning" title="DNS resolution issues on some subnets">
  Your network may force you to go through a specific DNS server, one that does not match the addresses listed in the host's `/etc/resolv.conf` (an override applied by NetworkManager or systemd-resolved). In that case, name resolution will fail inside the container (and you will not be able to install packages), because the override does not affect the container, which sticks to the `/etc/resolv.conf` file. You must either change the contents of `/etc/resolv.conf`, or pass the `--dns IP_DNS` argument when starting the container. To find `IP_DNS`, you can use `nmcli device show | grep DNS` or `resolvectl status`.
</Note>

## Creating images: the Dockerfile

### The efficient method

- A **Dockerfile** is a text document containing all the commands you would normally run manually on an OS (apt/yum, software configuration, user creation, environment variables…).
- With the `docker build` command, Docker creates images **automatically** by reading the instructions in the Dockerfile.
- For **each** instruction in the Dockerfile, `docker build` generates an image — and a layer if needed (since it may be empty). These are called **intermediate** images, except for the last instruction, which corresponds to the final image (the *top level image*).
- Intermediate images are generated by a **commit** mechanism applied to the containers in which the instructions were executed.
- You can create several images from a single Dockerfile (but you shouldn't).

<Note type="info" title="Evolution · the builder (BuildKit)">
  - **Before.** `docker build` used the legacy builder: one image (and one committed container) per instruction, strictly sequential steps — that is what this course's terminal outputs show.
  - **The problem.** No parallelism between independent instructions, a coarse cache, and secrets (keys, tokens) that ended up baked into layers if passed through an `ARG`.
  - **Today.** Since Docker 23.0 (2023), **BuildKit** is the default builder (`docker buildx`): a parallelized step graph, fine-grained cache (`RUN --mount=type=cache`), ephemeral secrets (`--mount=type=secret`), multi-architecture builds. The layers + cache model stays conceptually the same, but intermediate images no longer appear in `docker images -a`.
  - **Why it is better.** Faster, more reproducible builds, secrets kept out of the image, better cache granularity. To reproduce this course's historical outputs: `DOCKER_BUILDKIT=0 docker build .` (legacy builder, on its way out).
</Note>

### The notion of context

- You must **dedicate a directory** to each Dockerfile-based image build. Indeed, `docker build` works with a notion of **context** that applies recursively from the path of the directory containing the Dockerfile.
- Just like with git, you can use a `.dockerignore` file: `docker build` will ignore all the files and directories listed in it (they will not be part of the context).
- Comments in a Dockerfile are prefixed with `#`.

<Note type="danger" title="Never create a Dockerfile at the root of the system">
  The context sent to the Docker daemon would include the entire filesystem!
</Note>

### The instructions

You can use **environment variables** (such as `HOME`, `PATH`…) that are interpreted by certain instructions, marked ✦ below:

| Instruction | Role |
|---|---|
| `FROM` | base image |
| `RUN` | run a command (new layer) |
| `CMD` | default command of the container |
| `LABEL` ✦ | metadata (`MAINTAINER` is deprecated in favor of `LABEL`) |
| `EXPOSE` ✦ | document a listening port |
| `ENV` ✦ | define an environment variable |
| `ADD` ✦ / `COPY` ✦ | copy files from the context into the image |
| `ENTRYPOINT` | executable of the container |
| `VOLUME` ✦ | declare a mount point |
| `USER` ✦ | user for the following instructions |
| `WORKDIR` ✦ | working directory for the following instructions |
| `ARG` | build-time variable |
| `ONBUILD` ✦ | instruction deferred to child images |
| `STOPSIGNAL` ✦ | stop signal |
| `HEALTHCHECK` | health probe |
| `SHELL` | shell used by `RUN` instructions |

<Note title="Label using the OCI convention">
  Rather than ad-hoc `LABEL` keys (`maintainer`, `version`…), the ecosystem has standardized a set of **OCI** labels (`org.opencontainers.image.*`) that registries and tools know how to read:

  ```dockerfile
  LABEL org.opencontainers.image.title="sshd" \
        org.opencontainers.image.authors="firstname.lastname@example.com" \
        org.opencontainers.image.version="1.0.0" \
        org.opencontainers.image.source="https://github.com/example/sshd"
  ```
</Note>

### Example: the Dockerfile of an ssh server

`/home/jmc/sshd/Dockerfile`:

```dockerfile
# Base image
FROM debian:testing-slim

# Environment variables
ENV VERSION=1 \
    DEBIAN_FRONTEND=noninteractive

# Image metadata
LABEL maintainer="Jean-Mathieu Chantrein" \
    maintainer_email="jean-mathieu.chantrein@univ-angers.fr" \
    version="${VERSION}"

# Creating a regular user
RUN useradd lambda --create-home --shell /bin/bash && \
    echo "lambda:password" | chpasswd

# Installing packages, removing the apt cache
RUN apt update && \
    apt install -yq openssh-server && \
    rm -rf /var/lib/apt/lists/* && \
    mkdir /var/run/sshd

# setuid on the ssh daemon
RUN chmod u+s /usr/sbin/sshd

# root => lambda
USER lambda

# Default command when the container starts
CMD ["/usr/sbin/sshd", "-D"]
```

<Note title="ENV DEBIAN_FRONTEND: do not reproduce this">
  Declaring `DEBIAN_FRONTEND=noninteractive` as a **persistent** environment variable (`ENV` instruction) is bad practice — it will remain active in all containers and all child images ([GitHub issue](https://github.com/moby/moby/issues/4032)). Prefer the scope of a single command: `RUN DEBIAN_FRONTEND=noninteractive apt install …`. The [lab](../tp-docker-wordpress/) comes back to this.
</Note>

<Note type="info" title="Building for several architectures (buildx)">
  An image built on an x86 PC does not run as-is on a Raspberry Pi or an Apple Silicon Mac (ARM). BuildKit produces a **multi-architecture** image in a single command:

  ```bash
  docker buildx build --platform linux/amd64,linux/arm64 -t myimage:1.0 --push .
  ```

  The registry then stores a *manifest* that points each machine to the right variant: a plain `docker pull myimage:1.0` automatically fetches the local architecture.
</Note>

### A few more things

- It is important to tag/version your production images.
- The largest layers should be the lowest ones (efficient use of the cache mechanism).
- Instructions likely to change frequently should sit in the **upper** layers.
- Beware of `update` and `install` directives when using the cache: a `RUN apt update` served from the cache may be stale.
- Any change to the context (modification of a file added via `ADD` or `COPY`) triggers a rebuild of the images from the affected layer onward.
- Each build of a Dockerfile generates a **new** image (unless the cache is used entirely).
- Each instruction is **independent**: writing `RUN cd /tmp` makes no sense —

```console
Step 2/4 : RUN cd /tmp
 ---> Running in d172a17e8705
 ---> 1eb259884a9f
Step 3/4 : RUN pwd
 ---> Running in 80a06bfd0434
/
 ---> 9f59028b0464
```

- To change the directory for the following instructions (including for any child images), use the `WORKDIR` instruction.

<Note title="The entrypoint pattern of official images">
  `ENTRYPOINT` is also used to run a **bootstrap script** before the service. This is the pattern of official images: a `docker-entrypoint.sh` prepares the environment (permissions, configuration, first-run initialization), **drops root privileges** with `gosu` (or `su-exec`), then hands over to the process via `exec "$@"` (so it becomes PID 1). This is the **right** answer to "root in the container" — far cleaner than the lab's handcrafted `setuid`.

  ```dockerfile
  COPY docker-entrypoint.sh /usr/local/bin/
  ENTRYPOINT ["docker-entrypoint.sh"]
  CMD ["apache2", "-D", "FOREGROUND"]
  ```

  ```bash
  #!/bin/sh
  # docker-entrypoint.sh — prepare the environment, then drop privileges
  chown -R www-data:www-data /var/www/html
  exec gosu www-data "$@"
  ```
</Note>

## Multi-stage builds

Up to now, **everything used to build** the image ends up **in the final image**: compiler, development headers, downloaded archives, `wget`, `unzip`… all of it dead weight and **attack surface** that you will never run in production. The *multi-stage build* solves exactly this: you **build** in a throwaway stage, and **ship** only the useful artifacts in a lean final stage.

- A single Dockerfile can contain **several `FROM` instructions**: each one opens a new **stage**.
- You name a stage with `FROM image AS stage_name`.
- You copy a file from one stage into another with `COPY --from=stage_name /source /destination`.
- **Only the last stage** produces the final image; the intermediate stages — and all their tooling — are **discarded**: they contribute no layer to what ships.

<Note title="Three handy reflexes">
  - `COPY --from=` can also point at an **external image**: `COPY --from=nginx:1.27 /etc/nginx/nginx.conf /tmp/` grabs a file without installing anything.
  - `docker build --target stage_name` stops the build at a given stage — handy for debugging, or to produce a *dev* image more complete than the *prod* one.
  - The final image can start from a **minimal** base: `alpine` (≈ 5 MB), a *distroless* image, or even `scratch` — an **empty** image (zero bytes) — when a **static** binary is self-sufficient.
</Note>

```console
jmc@laptop ~ $ docker images
REPOSITORY   TAG     IMAGE ID      CREATED         SIZE
hello        latest  a1b2c3d4e5f6  2 seconds ago   892kB
debian       13      7b9b13f7b9c0  2 weeks ago     117MB
jmc@laptop ~ $ docker run --rm hello
Hello from a multi-stage build!
```

<Note type="info" title="Multi-stage and BuildKit">
  With **BuildKit** (the default builder since Docker 23), intermediate stages are **never** tagged nor listed by `docker images -a`: they live in the build cache, and **independent** stages can even be built **in parallel**. This is one of the cases where the legacy builder's "one intermediate image per instruction" model no longer applies.
</Note>

<Exercise id="multistage-scratch" level={2}>
  - Put both files above in the same directory.
  - Build the image (`docker build -t hello .`), run it, then note its size with `docker images`.
  - Build **only** the compile stage with `docker build --target builder -t hello-builder .`. Compare the two sizes: what do you notice?

  <Fragment slot="solution">
    ```bash
    docker build -t hello .
    docker run --rm hello                          # Hello from a multi-stage build!
    docker build --target builder -t hello-builder .
    docker images                                  # hello ≈ 1 MB, hello-builder several hundred MB
    ```

    The final image holds only the static binary; the `builder` stage, with its compilation toolchain, is hundreds of times heavier — which is exactly what the multi-stage build lets you **avoid** shipping.
  </Fragment>
</Exercise>

<Note title="What about our WordPress service?">
  WordPress is **interpreted** PHP: multi-stage is less spectacular there than for a compiled language. You could still **download and unzip** the sources in a *builder* stage, then copy only `/var/www/html` into the final image — which would then no longer need `wget` or `unzip`. The [lab](../tp-docker-wordpress/) installs them in the final image: a deliberate pedagogical trade-off, not a necessity.
</Note>
