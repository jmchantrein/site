### Context for using Docker

- In this course, we will use Docker exclusively on a **Linux kernel** (there are also native Windows containers, for containerizing Windows applications on Windows Server).
- You can use Docker natively on a GNU/Linux OS.

## Definitions and prerequisites

### Images and layers: the foundations of Docker containers

- An **image** refers to an **ordered** collection (by stacking) of **modifications** of a root filesystem.
- Each of these filesystem modifications is kept in **a layer**, identifiable by the SHA-256 hash of its content. An image therefore refers to an **ordered** collection of layers.
- Each **intermediate** layer is associated with an **intermediate** image.
- Layers are created by running containers that are then turned into images (don't panic, the slideshow below shows this step by step).
- There can be empty layers (*empty layers*).
- The layers referenced by an image are necessarily **read-only**.
- An image is **unique**: it is identified by a SHA-256 hash of its descriptor. It **has no state** (you don't run an image) and it is **immutable**.

### A container: the instantiation of an image

- The container's root filesystem **is** the root filesystem of the underlying image, **but** it is read-only.
- We have **the illusion** of modifying the filesystem inside the container, because the changes are made in **a new read/write layer** (*Copy On Write* technology, COW).
- Deleting the container **always deletes this read/write layer**.
- You can make the modifications made in a container's filesystem permanent by creating a **new** image, which adds the newly created layer to the read-only layers: the layer that was read/write becomes read-only.
- This is done with the `docker commit` command.

<Note type="danger" title="What you really need to understand">
  - Image ≠ layer ≠ container.
  - A layer contains **only** the **differences** of a filesystem.
  - A **set** of layers can form a root filesystem.
  - Layers know nothing about the images that use them.
  - Layers can be used by **several** images.
  - An image can reference **more than one** layer.
  - An image is something you can instantiate. The **result** of this instantiation is a container.
  - When a container starts, certain pieces of information from the image used to instantiate it allow its root filesystem to be built.
</Note>

  </Slide>
  <Slide>
    #### Two images, n shared layers

    <LayerStack state="partage" />

    Advantage: images x and y have n layers in common (**resource sharing**). Each locally built layer has a corresponding image: this is called an **intermediate image** (`docker images --all`).
  </Slide>
  <Slide>
    #### Deleting an image

    <LayerStack state="partage" />

    When you delete an image:

    - if the layers are used by other images, they will not be deleted;
    - otherwise, there is a cascading destruction of the intermediate parent images and their associated layers;
    - deleting x (resp. y) does not prevent the use of y (resp. x).
  </Slide>
  <Slide>
    #### The image architecture: a tree

    The overall image architecture can be visualized as a **tree**: the leaves are images without children, the nodes correspond to non-intermediate images (normally tagged). These leaves and nodes are the images intended to be instantiated in production.

    - We say that image y **has image x as its parent**.
    - A **top level image** that has no tag (`<none>`) is said to be *dangling*.
    - A top level image should always be tagged.
    - When designing a local image architecture, you should **maximize the number of shared layers and minimize the number of base images**.
  </Slide>
</Slides>

<Note title="Special images">
  - A *top level image* is not an intermediate image.
  - An image tagged `<none>` is an intermediate image; otherwise it is a dangling top level image.
  - A layer marked `<missing>` in `docker history` is not an error: since 1.10, Docker keeps no **local intermediate-image ID** for the layers of an image that was pulled (`docker pull`) or built elsewhere — you do have the layer, but no image ID to attach to it.
</Note>

<Note type="warning" title="Before version 1.10 (historical)">
  Each image was identified by a random number and associated with **a single** layer, identified by the same identifier as the image — hence security issues caused by the random number. More information: [Explaining Docker image IDs](https://windsock.io/explaining-docker-image-ids/).
</Note>

## A few basic commands

### Image management

- `docker pull`: download an image from a registry ([Docker Hub](https://hub.docker.com) by default);
- `docker push`: push (upload) an image to a registry;
- `docker search`: search for an image on a registry;
- `docker images`: list the images present locally;
- `docker history`: show the creation history of an image;
- `docker rmi`: delete an image;
- `docker commit`: build an image from a container;
- `docker build`: build an image from a Dockerfile.

<Note type="info" title="Evolution · image registries">
  - **Before.** `docker pull`/`push` targeted the **Docker Hub** by default, with no real limit: you pulled and pushed freely.
  - **The problem.** Since November 2020, the Docker Hub enforces **pull quotas** (per IP for anonymous users, per account otherwise) — a CI build can be interrupted by a *rate limit*.
  - **Today.** The image format and the *registry* protocol are **standardized (OCI)**: you authenticate (`docker login`) and spread your images across several registries — Docker Hub, **GHCR** (`ghcr.io`), **Quay**, or a self-hosted private registry.
  - **Why it is better.** No more dependence on a single provider, controlled quotas, and images interoperable from one registry to another thanks to the OCI standard.
</Note>

### Container management

- `docker run`: start a container;
- `docker stop` / `start` / `restart` / `pause` / `unpause`: stop / start / restart / pause / resume a container;
- `docker logs`: get a container's logs;
- `docker kill`: kill a container;
- `docker rm`: delete a container;
- `docker ps`: list running containers.

<Note title="Several commands for the same result">
  The commands above are Docker's historical commands. Over time, Docker has grouped its commands by the "objects" they concern: containers under `docker container <cmd>`, images under `docker image <cmd>`, and so on for `config`, `network`, `node`, `plugin`, `secret`, `service`, `stack`, `swarm`, `system` and `volume`. But `docker container ls -aq` **is identical to** `docker ps -aq`.
</Note>

### Docker management

- `docker info`: display information about the Docker ecosystem on the machine;
- `docker inspect`: display low-level information about Docker objects (images, containers, networks…);
- `docker version`: display the Docker version.

### Docker "philosophy": recommended best practices

- A container should be ephemeral (separation and mount point of data inside the container).
- One container = 1 (micro)service (= 1 process for the purists).
- You should not be root inside a container (sshd, crond, firewall… belong to the host).
- If you ever need to be root: use setuid, gosu or su-exec.
- Minimize image size (no unnecessary extras).
- Minimize the number of layers in an image (Dockerfile: multi-line arguments, pipes).
- Maximize layer sharing.
- Put the largest layers closest to the root (taking advantage of the `docker build` cache).

### Docker "philosophy·ies"?

- Adapt the best practices to your own needs and usage.
- For more than one process per container, look into supervisord.
- Are you using the right containerization technology?

### A few easy cases

<Exercise id="cas-faciles" level={1}>
  Take the time to launch a few containers, simply:

  1. Run `/bin/bash` in an interactive ubuntu container.
  2. Run `/bin/ls` in a non-interactive ubuntu container.
  3. Run `/bin/ps -eflH` in a non-interactive ubuntu container.
  4. Run `/bin/bash` in an interactive ubuntu container named `mon_conteneur_c0`.
  5. Detach from the `mon_conteneur_c0` container with Ctrl+p Ctrl+q.
  6. Reattach to the `mon_conteneur_c0` container.
  7. Exit the `mon_conteneur_c0` container.
  8. List the running containers, then list all containers.
  9. Delete all stopped containers with a single command (`docker container help`).

  <Fragment slot="solution">

    2. `docker run ubuntu /bin/ls`
    3. `docker run ubuntu /bin/ps -eflH`
    4. `docker run -it --name mon_conteneur_c0 ubuntu`
    5. Ctrl+p Ctrl+q
    6. `docker container attach mon_conteneur_c0`

### A few tricky cases

<Exercise id="cas-penibles" level={2}>
  You will use the `docker run` and `docker container` commands (`docker <cmd> help` for help on the `<cmd>` command):

  1. Try to run `/bin/bash` in an interactive alpine container.
  2. Run `echo hello world !` in a non-interactive centos container `c1`.
  3. Try to start the `c1` container once. Run `docker logs c1`.
  4. Run `/bin/bash` in a non-interactive centos container `c2`.
  5. Try to start the `c2` container once. Run `docker logs c2`.
  6. Run `/bin/bash` in an interactive centos container `c3`, then exit c3 with `exit`.
  7. Start the `c3` container, connect to it with a new bash, then detach from it.
  8. List the processes of the `c3` container from the host **and** from inside the container (`ps`).

  <Fragment slot="solution">
    1. `/bin/bash` is not present by default in alpine. The underlying idea: minimize image size — you only install what you need; the Alpine distribution is just 5 MB!
    2. `docker run centos echo hello world !`
    3. `c1` is in non-interactive mode and `echo` has no event loop; but `hello` is actually printed twice (look at the logs).
    4. `c2` is in non-interactive mode and no pseudo-terminal is allocated.
    5. The `/bin/bash` command is actually executed twice.
    6. `docker run -it --name c3 centos /bin/bash ; exit`
    7. `docker start c3 ; docker exec -it c3 /bin/bash` then Ctrl+p Ctrl+q.
    8. `docker container top c3` (PIDs as seen from the host); `docker exec c3 ps` (PIDs as seen from the container).
  </Fragment>
</Exercise>

## Data volatility, persistence and security

Let's go a bit further (*something more ambitious*): what happens if we destroy the filesystem **inside** a container?

```console
jmc@laptop ~ $ docker run -it ubuntu /bin/bash
root@dfdf734253a9:/# ls
bin  boot  dev  etc  home  lib  lib64  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp  usr  var
root@dfdf734253a9:/# rm -rf /     # No impact on the host: no volume mounted
rm: it is dangerous to operate recursively on '/'
rm: use --no-preserve-root to override this failsafe
root@dfdf734253a9:/# rm -rf --no-preserve-root /
...
root@dfdf734253a9:/# ls
bash: /bin/ls: No such file or directory
root@dfdf734253a9:/# exit
```

The container is destroyed from the inside… but the host is unharmed, and so is the ubuntu **image**: only the container's read/write layer was ravaged. The proof — a new container from the same image is intact:

```console
jmc@laptop ~ $ docker rm dfd
dfd
jmc@laptop ~ $ docker run -it ubuntu /bin/bash
root@63f0abcc808a:/# ls
bin  boot  dev  etc  home  lib  lib64  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp  usr  var
root@63f0abcc808a:/# exit
```

However, if we **mount** the host's root filesystem inside the container, the danger becomes real:

```console
jmc@laptop ~ $ docker run -it -v /:/danger_racine_hote --hostname myhostname --name myname ubuntu /bin/bash
root@myhostname:/# ls /danger_racine_hote
bin  boot  cdrom  dev  etc  home  initrd.img  lib  lib64  lost+found  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp  usr  var  vmlinuz
root@myhostname:/# # Do NOT run rm -rf /danger_racine_hote
```

Finally, the `--rm` option automatically deletes the container (and its RW layer) on exit:

```console
jmc@laptop ~ $ docker run -it --rm ubuntu /bin/bash
root@30481297b94c:/# exit
jmc@laptop ~ $ docker ps -a
CONTAINER ID  IMAGE  COMMAND  CREATED  STATUS  PORTS  NAMES
```

<Note type="success" title="Summary">
  - Running containers that contain root processes can be dangerous.
  - You can mount host directories inside the container to ensure the persistence of certain data: this technology is called **volumes** (*data volumes*).
  - For mounts on shared resources (NFS, iSCSI, FC): see the *docker volume plugins*.
  - There is a persistence technology that works across containers: *data volume containers*.
</Note>

## Container security

<Note type="warning" title="Namespaces: the basic isolation">
  - Processes running in a container can neither see nor affect processes running in another container or in the host system.
  - Each container has its own network. If the host is configured accordingly, containers can interact with one another (and with the host) through their respective network interfaces.
  - All containers on a Docker host are connected to *bridges* (virtual switches).
  - It is still inadvisable to run root applications (ssh, vpn, dhcp…): the host is supposed to provide for these needs.
  - If you want to change the default isolation behavior, this requires hands-on work and knowledge of *capabilities* — that is a price to pay when sharing the same kernel.
  - More information: [Docker security](https://docs.docker.com/engine/security/).
</Note>

<Note type="warning" title="Capabilities">
  - Running containers that contain root processes can be dangerous.
  - In case of privilege escalation from the container to the host: by default, root inside the container does not have the same rights as a potential root on the host.
  - Indeed, it does not have all the *capabilities*.
  - More information: [man capabilities](https://linux.die.net/man/7/capabilities).

  Capabilities granted by default: `CAP_CHOWN`, `CAP_DAC_OVERRIDE`, `CAP_FSETID`, `CAP_FOWNER`, `CAP_MKNOD`, `CAP_NET_RAW`, `CAP_SETGID`, `CAP_SETUID`, `CAP_SETFCAP`, `CAP_SETPCAP`, `CAP_NET_BIND_SERVICE`, `CAP_SYS_CHROOT`, `CAP_KILL`, `CAP_AUDIT_WRITE`.
</Note>

<Note type="warning" title="cgroups: resource limiting">
  - Per-container resource accounting and limiting.
  - By default: a fair share of resources (CPU, RAM, I/O) per container.
  - Prevents a single container from using all the resources (demo: a *fork bomb* confined with `--cpuset-cpus 0`).
  - In a way, they prevent certain denial-of-service attacks.
  - "Guarantees" nearly constant availability and performance.
</Note>

<Note type="danger" title="Docker daemon: attack surface">
  - The Docker daemon runs as root!
  - Only an administrator should be able to control this daemon.
  - How much do you trust the Docker image you are downloading?
  - It is recommended to run only Docker on a server (apart from ssh, dhcp, etc.): if there are other applications, they should be containerized or moved elsewhere.
  - Do not test containers on a production machine!
  - Since Docker 20.10, a **rootless** mode allows the daemon and containers to run without root privileges (this is also Podman's natural mode of operation).
</Note>

## Graphical containers

- It is entirely possible to run graphical containers (a browser in a container, for example).
- In general, you map the host's `/tmp/.X11-unix` and `~/.Xauthority` into the container with Xorg; under Wayland, you share the compositor's `wayland-0` socket.
- Some people use VNC (much less efficient, but more secure…).
- We will not cover this aspect any further in this course.
